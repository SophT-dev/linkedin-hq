# Sub-skill: hook selection

Read by `linkedin-post/SKILL.md` Phase 3. Job: pick or adapt a real hook for the post being
drafted — never invent one from nothing.

**Standing rule (Sophiya's call, 2026-07-08): no original hooks, full stop, for now.** Every
hook must trace to a real Template Library row or a named F-format swipe example. Not "inspired
by" — actually built by adapting a specific real post. If you can't point to the exact source
row, it's not ready to show her. This is deliberate: prove the templates work before earning the
right to improvise.

## Where to pull from, in order

1. **The assigned source** — whatever Phase 1 picked (a Content Calendar slot, a Post Ideas row,
   a Domain Synthesis doc, or a WinsLog entry). This determines the topic/angle; it does NOT by
   itself give you a hook structure.
2. **Template Library tab** — filter to the same `domain` (or a close one) as the topic, sort by
   `likes` and separately by `comments`. Pick the top 2-3 candidate hooks whose *structure* fits
   the angle (a big-number hook for a stat, a contrarian hook for a hot take, etc.) — read
   `playbook/FORMAT-LIBRARY.md`'s F1-F12 definitions to match structure to angle, not just vibes.
3. **Adapt, never copy.** Swap in Bleed AI's own specifics (a real WinsLog number, a real tool,
   Taha's or Sophiya's actual voice) for whatever the source post used. If you can't adapt a hook
   without keeping someone else's specific phrasing, pick a different one.
4. **The Copywriting Bible** — `playbook/COPYWRITING-BIBLE.md` Part A§3 + B§1 + the Part C swipe
   file is the craft layer underneath any hook: what makes an opener actually pull (specific,
   short, a curiosity gap or a stated benefit — never a rhetorical question). Run every candidate
   hook through the Bible's opener test before presenting it.
5. **The curiosity-gap test (Part F§3, added 2026-07-08)** — a flat statement of the topic isn't
   enough. Before showing a hook, confirm it actually opens a gap: a Conflict or Unusual framing
   (a trusted assumption vs. a specific contradicting fact), second person, a real number, and the
   gap gets resolved starting in the very next lines, never left dangling past the first section.
   A hook that only informs reads flat; one that contradicts what the reader already believes
   forces the "wait, what?" reaction. Run this test on every candidate, not just the one you pick.

**Critical: Part F is a lens for picking and sharpening step 2's real candidate, never a license
to write from scratch (Sophiya's correction, 2026-07-08).** The order is always: pull 2-3 real
candidate rows from the Template Library first (step 2), THEN use Part F's curiosity-gap test to
decide which candidate's structure fits best and how to tighten the adaptation (tenser phrasing,
sharper contrast, tighter char count). If a "better" hook occurs to you that isn't a tightened
version of an actual row you can point to, it's an original hook and it's not allowed — go back
to the Template Library and find the row that already has the shape you're reaching for. The test
in step 5 evaluates and improves an adaptation; it does not generate one.

## Hard character limit — check this on every hook, every time

LinkedIn truncates the feed preview behind a "...see more" button at roughly 140 characters on
mobile and 200-250 on desktop (both third-party observed, not LinkedIn's own published spec —
see the pre-publish checklist in `COPYWRITING-BIBLE.md` for sources). **Count the actual hook
text — including spaces, line breaks, and emoji (standard emoji ~2 chars, flag/skin-tone emoji
4-7 chars) — and keep it at 120-130 characters or under.** If a hook you've adapted from a
Template Library row runs long, cut it — don't ship a hook that gets chopped mid-word on a phone.

## What to cite back to the orchestrator

For each draft, hand back: the hook text, which Template Library row (or F-number format) it was
built from, and a one-line "why this structure fits this angle." This is what Phase 3 uses to
satisfy philosophy #2 (show the work) — every draft must be able to say what it's based on.

## Guardrail

If nothing in the Template Library fits the assigned topic/domain well, say so explicitly rather
than forcing a mismatched hook — fall back to a Domain Synthesis doc's own framing, or flag it
back to the orchestrator so Sophiya can pick a different source for this slot.

## Hooks don't stand alone — see `post-structure-selection.md`

A hook is the opening line; it doesn't determine how the *rest* of the post looks (line length,
white space, bullets vs. arrows, brick vs. broken). That's a separate decision — read
`post-structure-selection.md` in this same folder right after this one, every time.
